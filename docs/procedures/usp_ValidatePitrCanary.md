# cfg.usp_ValidatePitrCanary

> *Validation Layer - Validation*

## Overview

`cfg.usp_ValidatePitrCanary` is responsible for validating the logical correctness of a restore scenario using deterministic canary records.

It compares the expected `BEFORE` / `MARK` / `AFTER` evidence between the source database and the restored target database in order to confirm whether the recovery boundary was respected.

This procedure does not perform restore operations. Its purpose is to determine whether the restored database matches the expected logical state after a recovery test.

## Responsibilities

- Validate logical recovery correctness using canary evidence  
- Compare source and target canary states  
- Confirm that the recovery boundary was respected  
- Produce structured validation results  
- Support deterministic evidence for STOPBEFOREMARK scenarios  
- Provide validation output suitable for telemetry persistence  

## Parameters

| Parameter | Type | Description |
|----------|------|-------------|
| @SourceDB | SYSNAME | Source database where the original canary records were generated. |
| @TargetDB | SYSNAME | Restored target database to be validated. |
| @Token | NVARCHAR(128) | Identifier of the canary record. |

## Execution Flow

The procedure follows a deterministic validation pattern:

1. Inspect source database canary state  
2. Inspect restored target database canary state  
3. Compare `BEFORE` / `MARK` / `AFTER` expectations  
4. Determine whether validation was executed successfully  
5. Determine whether the validation passed  
6. Return a structured validation contract  

## Example Usage

```sql
EXEC cfg.usp_ValidatePitrCanary
    @SourceDatabase = 'AdventureWorks2022',
    @TargetDatabase = 'AdventureWorks2022_RestoreTest',
    @Token = '10C773962AC24102B0762D358633CA45';
```
## Outputs

The procedure generates outputs at two levels.

### *1. Structured result set*

The procedure returns a validation contract containing:

- source and target database names
- canary identifiers
- source canary evidence flags
- target canary evidence flags
- `CanaryValidated`
- `CanaryPassed`
- `CanaryMessage`

This contract is intended to be consumed by the orchestration layer and persisted into `[log].[RestoreTestRun]`.

### *2. Runtime validation trace*

When executed in operational workflows, this procedure may also contribute validation messages to the runtime output generated by the calling orchestration procedure.

These messages provide visibility into:
- validation completion
- pass/fail state
- logical recovery outcome

The authoritative validation evidence remains the structured contract returned by this procedure and the telemetry persisted by the caller.

## Related Components
- `[cfg].[usp_RunRestoreTests]` → Recovery orchestration engine
- `[cfg].[usp_RestorePointInTime]` → Restore execution engine
- `[dbo].[PitrCanary]` → Logical validation artifact table
- `[log].[RestoreTestRun]` → Stores validation evidence at header level

## Design Notes

This procedure represents the logical validation layer of the framework.

Its purpose is to verify recoverability using semantic evidence rather than relying only on restore completion status. By validating BEFORE / MARK / AFTER canary states, the procedure provides deterministic proof that the restore stopped at the intended logical boundary.
